# spec-conformance

仕様書（`doc/`・`spec/`・RFC・OpenAPI スキーマ・ADR など、そのプロジェクトが正典と定めた文書群）を基準として、実装との乖離を **バッチ単位のマルチエージェント Workflow** で体系的に埋める。1バッチ = 全テスト緑・不変条件維持・コミット可能な単位。実装ステージには必ず**独立した監査ステージ**を組み合わせる。

## このスキルの分業

**プロジェクト非依存の手順**は、このスキル・`~/.claude/agents/spec-*.md`・`~/.claude/workflows/spec-conformance-phase.js` に置く。**プロジェクト固有の事情**（正典の場所・テストの回し方・壊してはいけない中核契約・1フェーズの単位）は `.claude/conformance.json` と、そこから組み立てる `commonRules` 文字列だけに閉じ込める。

このファイルやエージェント定義に特定プロジェクトの前提（そのプロジェクトのテストランナー名・ドメイン用語・過去の作業の逸話）を書き足さないこと。別プロジェクトから読んだときに、無関係な前提を正しい指示だと誤解する。

## 0. 起動時にやること

1. プロジェクトルートの `.claude/conformance.json` を読む。無ければ同梱の `conformance.template.json` を雛形に、`specGlobs` / `testCommand` / `fullSuiteCommand` / `invariantChecks` / `heavyTests` をユーザーと確定してから作る。**推測で埋めない** — 誤ったテストコマンドは以降の全バッチの検証を静かに無効化し、赤を緑と誤認したままコミットが積み上がる。よくあるスタックの初期値は `reference/stack-presets.md` にある。
2. `CLAUDE.md` などのプロジェクト方針を読み、`commonRules`（全エージェントのプロンプト冒頭に付く共通前文）に織り込む。`commonRules` に必ず入れるもの: リポジトリのパス／正典の場所と読み方／**中核契約**（壊れたら実装をやり直す性質。`coreContracts`）／テスト方針（個別テストのみ・フルスイート禁止）／git 操作の禁止／コメントを自己完結させるルール。
3. 乖離の一覧（監査レポート・ロードマップ）が既にあればそれを入力にする。無ければ先に作る（次節）。

## 0b. 乖離一覧が無い場合（Discovery）

正典を章・ファイル単位に割り、read-only エージェントを並列に立てて「この章が要求していることのうち、コードに存在しない／食い違うもの」を列挙させる。一次検出は必ずノイズを含む（別名・別経路で既に満たされている、正典の記述が古い）ので、**検出された各乖離を別のエージェントが独立に再確認**してから一覧に入れる。

一覧は `roadmapFile` に書き出し、項目ごとに「安定した ID・正典の節参照・要求の要約・関係するコードのパスとシンボル名」を持たせる。行番号は最初のコミットで陳腐化するので、ID とシンボル名を主キーにする。

## 1. 依存順にフェーズへ分ける

全乖離を依存関係に沿ったフェーズ（基盤 → それに依存する層）へ並べる。1フェーズは 8〜20 件程度で、それ自体がコミット可能な単位になるように切る。フェーズ内では、**同一ファイル群を触る項目をクラスタにまとめる**（クラスタは逐次実装してファイル競合を避ける）。

高リスク群（初期化・データ移行・数値較正・永続表現の変更など、壊れたときに原因の切り分けが難しい領域）は、API 配管や表示層のような低リスク群と**別フェーズに分離**する。混ぜると、赤が出たときにどちらが原因か分からなくなる。

## 2. 各フェーズを Scope → Implement → Conform → Repair で回す

`Workflow({ name: "spec-conformance-phase", args })` を起動する。Workflow スクリプトはファイルシステムへアクセスできないので、設定値は必ず `args` 経由で渡す（`commonRules` / `scopeContext` / `items` / `clusters`、任意で `model` / `wiringRule` / `testRule` / `auditFocus`）。使う agentType は `spec-scout` / `spec-implementer` / `spec-auditor`。

- **Scope（並列・read-only）**: `spec-scout` が項目ごとに正典の原文と現行コードを読み、方針・リスク・「既に解消済みか」を確定する。式・列挙・順序・既定値は原文どおり写させる。
- **Implement（クラスタ別に逐次）**: `spec-implementer` が仕様どおり実装する。自分が触ったファイルに対応するテストだけ smoke で回してよい。
- **Conform（並列・read-only）**: `spec-auditor` が**実装者の報告を一切見ずに**、正典とコードだけで DONE/PARTIAL/NOT_DONE を再判定する。これがこの手順の肝である。
- **Repair（既定 ≤2ラウンド）**: 未達項目を修正し、同じ監査観点で再判定する。

Scope の結果が大きい場合は一時ファイルへ書き出し、実装者にはパスを渡す（原文に根ざした仕様を読ませるため）。

エージェントのモデルは既定でセッションのモデルを継承する。安価なモデルで回したい場合だけ `args.model` を指定する。

## 3. 検証はメインループ（あなた）が所有する

**Workflow のエージェントにテストスイートを実行させない。** 並列エージェントが同時にテストを起動すると CPU を奪い合い、1本あたりの実行時間が数倍に伸びる。伸びた結果ハーネスのプロセス寿命やタイムアウトに掛かって打ち切られ、「本当に落ちた」のか「打ち切られた」のか区別できない赤だけが残る。並列化が効くのは、CPU をほとんど使わずレイテンシが支配的な作業（コードを読む・書く・監査する）に限られる。テスト実行は本質的に直列（あるいはテストランナー自身の並列オプション）で回すべき仕事である。

エージェントに許すのは「自分が触ったファイルに対応するテストを1ファイルずつ」まで。Workflow が返ったら、**あなたが**次を順に実行してからコミットする:

1. 同梱の hygiene-check で、リポジトリだけを見る第三者が辿れない参照（作業メモへのパス、監査所見の採番など）を掃討する。設定はそのまま渡せる: `python ~/.claude/skills/spec-conformance/hygiene-check.py --config .claude/conformance.json`（Windows なら `powershell -NoProfile -ExecutionPolicy Bypass -File ...\hygiene-check.ps1 -Config .claude\conformance.json`）。両版は同一の判定を返す。
2. `fullSuiteCommand` でフルスイート。落ちたら、赤かったファイルを個別に回して切り分ける。
3. `invariantChecks` を実行する（`coreContracts` を機械的に検証するコマンド群）。
4. `heavyTests` を分けて回す。

**長時間コマンドの掟**: 起動したプロセスは起動元セッションが終われば道連れに死ぬ。重い検証は1回に1本だけ起動し、完走を見届けてから次へ進む（並行起動やシェルループでの連続起動は途中で打ち切られやすい）。ランナーの並列オプション（`pytest -n auto`、`jest --maxWorkers`、`go test -p` 等）を使うのはフルスイートの時だけにし、失敗の切り分けと重量テストは直列・単発で回す。

全部緑になって初めて、そのフェーズを `commitStyleSkill` でコミットする。

## 4. 判断のルール

- **仕様優先。仕様を削って実装に合わせない。** 削ってよいのは仕様の**内部矛盾**を根拠つきで示せる時だけ（他の節・他の実装と突き合わせ、孤立して食い違う1箇所を特定する）。既定値や式の典拠が別節を引いているのに、その別節が正反対を言っている、といった形で示す。
- **「本質的性質」と「実装手段の記述」を切り分ける。** 仕様が手段レベル（この関数を使う、このエンドポイントを共有する）で書いていても、本質は別のところ（特権の不在・共通の検証経路・単一の真実源）にあることが多い。本質を満たす実装を出し、手段の記述だけを仕様側で直す。手段どおりの実装が中核契約やスケールを壊す場合は、コストを**数字で**示す（憶測でなく、実際の既定値・データ量・関数実装を読んで）。
- **監査だけでも、テストだけでも、片方をすり抜ける。** 監査を通ってテストで落ちる（変更の副作用でテストの前提が古くなった）ことも、テストを通って監査で落ちる（仕様に届いていない・誤った正当化コメントが付いている）こともある。両方を必ず課す。
- **テストの期待値を変えるのは、実装が仕様に合った結果としてテストの前提が古くなった時だけ。** 正典の原文を引いて理由を書けること。テストを通すために実装を仕様から外さない。
- **横断的な配線を漏らさない。** 新しいフィールド・状態・列挙値・エラーコードは、それが通るべき全経路（永続化と読み戻し、スキーマとバリデーション、API 応答、移行、型定義、クライアント側、監査ログやハッシュ）に配線する。片側だけの実装はテストが緑でも仕様未達であり、これが最も再発しやすい欠陥である。

## 5. spec-auditor に探させる既知の失敗パターン

（agent 定義に埋め込み済みだが、レポートには毎回この観点で判定させる）

- 定数・タグ・フラグの書き換えだけで、仕様が要求する**実行順序や実挙動**は変わっていない（しばしば誤った正当化コメントが添えられている）
- 列挙値・フィールドを宣言しただけで、それを**生成・発火する経路が無い**デッドコード
- 仕様の式に似ているが係数・項が違う近似式
- 片側だけの実装（書き込み側のみで読み取り側が旧経路のまま、サーバ側のみでクライアント側が未対応、など）
- 新規フィールドが永続化・スキーマ・移行などの横断経路に配線されていない
- 仕様を「実装が楽だから」削ったが、内部矛盾という正当な根拠を欠く
- コメント・docstring に、リポジトリだけでは辿れない参照が混入している

自己申告が語れるのは「実装したこと」までで、「仕様に届いたか」は語れない。上のパターンはいずれも、実装者が誠実に「完了」と報告した状態で残りうる。だから監査は、実装者の報告を読まない別のエージェントが行う。

## 6. 別プロジェクトへ持ち込むときに決めること

1. **正典はどれか** — `specGlobs`。複数ある場合は優先順位も（例: OpenAPI スキーマ > 散文の設計書）。
2. **中核契約は何か** — `coreContracts`。壊れたら実装をやり直す性質を散文で列挙する（例: 同一入力での再現性、データ整合性、後方互換な永続形式、性能予算、認可境界）。これが `commonRules` に入り、scout のリスク評価と auditor の判定基準になる。
3. **どう検証するか** — `testCommand` / `fullSuiteCommand` / `invariantChecks` / `heavyTests`。`invariantChecks` が空のプロジェクトでは中核契約の回帰を機械的に捕まえられない旨をユーザーに伝え、最低1本作るよう提案する。
4. **何を書いてはいけないか** — `hygienePatterns` / `hygieneRoots`。プロジェクトの方針次第で、`#123` のような裸の採番を禁じる場合もあれば、課題管理システムへの参照として推奨される場合もある。既定は禁じない（`--ban-bare-hash-numbers` で有効化）。
5. **カスタム agentType と名前付き Workflow がグローバル `~/.claude/` から解決されるか** — 解決されない環境では、そのプロジェクトの `.claude/agents/` と `.claude/workflows/` へコピーする。最初は小さなフェーズを1つ通して解決経路を確認する。

設定ファイルの雛形は同梱の `conformance.template.json`、スタック別の初期値は `reference/stack-presets.md` にある。
